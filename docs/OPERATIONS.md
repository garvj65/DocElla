# Operations

This runbook covers the DocElla 1.0.0 single-service production architecture.

## Health

Use `GET /api/health` for readiness and liveness checks. A successful response includes:

- `service`: `DocElla API`
- `status`: `ok`
- `version`: `1.0.0`

The response is non-cacheable and contains no process, host, provider, secret, or filesystem data.

## Request IDs

Every API response includes `X-Request-Id`. JSON envelopes include the same value in
`meta.requestId`. Clients may provide a safe request ID containing letters, numbers, `.`, `_`, or
`-`, up to 128 characters.

Use request IDs rather than document contents when correlating a frontend error with backend logs.

## Structured logs and redaction

Logs are JSON and include safe operational fields such as:

- Event name
- HTTP method
- Route path without query string
- Status code
- Duration
- Request ID
- Safe provider error class and mapped code
- Generated byte count, schema ID, template ID, and flatten state

Logs intentionally exclude:

- Request and response bodies
- Authorization, cookie, and API-key headers
- Query strings
- Uploaded PDF bytes and filenames
- Extracted PDF text
- Groq prompts and completion bodies
- Extracted or reviewed values
- Generated PDF bytes
- Template filesystem paths and PDF field names

## Rate limits

Extraction and generation have independent limits.

Defaults:

- Extraction: 10 requests per 60 seconds.
- Generation: 20 requests per 60 seconds.

The limiters use process memory. They are suitable for one service process but are not shared across
multiple replicas. A distributed limiter is required before scaling horizontally with strict global
quotas.

Generation rate limiting runs before JSON parsing, validation, template loading, and PDF generation.

## Proxy trust

`TRUST_PROXY_HOPS=0` is the secure default. Set a positive bounded hop count only when every proxy in
the chain is trusted and understood. The configured value affects effective client IPs and therefore
rate limiting.

## Provider behavior

Groq is used only for extraction. PDF generation does not contact Groq.

Relevant settings:

- `GROQ_MODEL`
- `GROQ_TIMEOUT_MS`
- `GROQ_MAX_RETRIES`
- `GROQ_MAX_INPUT_CHARACTERS`

Provider timeout, rate-limit, unavailable, rejected, and invalid-output failures map to stable safe
error codes. Raw provider response bodies are not returned or logged.

## Graceful shutdown

On `SIGINT` or `SIGTERM`, the service:

1. Stops accepting new connections.
2. Closes idle connections.
3. Allows active requests to finish.
4. Waits up to `SHUTDOWN_TIMEOUT_MS`, default 10 seconds.
5. Force-closes remaining connections when the grace period expires.

A forced or failed shutdown sets a nonzero exit code. Repeated signals do not start duplicate
shutdown flows.

## Memory-only processing

Uploaded PDFs are held in memory by Multer. Extracted text, provider inputs, extraction values,
reviewed values, and generated PDF bytes are not persisted by DocElla.

The hosting platform may still capture process memory in crash dumps or snapshots according to its
own policy. Disable unnecessary diagnostics and restrict platform access.

## Safe diagnostic procedure

1. Collect the UTC time, route, safe error code, and request ID.
2. Check service health and recent deployment status.
3. Search structured logs by request ID.
4. Check rate-limit and provider availability events.
5. Reproduce with a synthetic document containing no real personal data.
6. Verify template structure and release gates when generation is affected.
7. Do not request the user's PDF, extracted text, form values, or API key unless a formal secure
   support channel is established.

## Incident response

For suspected key exposure:

1. Rotate the Groq key immediately.
2. Replace the hosting secret.
3. Restart the service.
4. Inspect repository history, CI logs, image history, deployment configuration, and provider logs.
5. Revoke the old key.
6. Run a synthetic extraction smoke.

For suspected document exposure:

1. Restrict service access.
2. Preserve safe request metadata and platform audit logs.
3. Review platform-level logging, tracing, snapshots, and crash dumps.
4. Confirm DocElla application logs and storage contain no document values or bytes.
5. Follow applicable legal and organizational incident procedures.

## Known v1 operational limitations

- No authentication or tenant isolation.
- No OCR.
- No persistence or resumable jobs.
- No distributed rate limiting.
- No queue for long-running extraction.
- No source coordinates or highlighting.
- No application telemetry or analytics.
- Standard PDF appearance font only.
